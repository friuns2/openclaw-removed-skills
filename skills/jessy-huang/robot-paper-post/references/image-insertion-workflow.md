# 配图工作流说明

## 设计目标

把“机器人论文写作 + 配图插入”做成稳定可复现流程，且默认不依赖额外安装的浏览器自动化环境。

## 默认策略

### 1. 先判断图的来源

优先顺序：
1. arXiv HTML 的官方 figure 图
2. 项目主页中的独立图片资源
3. 项目主页整页截图
4. 视频片段截图 / GIF

默认优先使用前两种，因为它们最稳定、最轻量、最容易复现。

### 2. 先判断是否必须下载到本地

- 只要求 Markdown 可预览：直接使用官方远程图 URL
- 要离线交付：下载到文章目录或 `paper_imgs/`
- 要发公众号后台：优先准备本地图

### 3. 图文映射规则

| 正文段落 | 推荐图片 |
|---------|---------|
| 为什么之前方法不行 | 范式对比图 / 问题设定图 |
| 模型如何设计 | 架构图 / 模块图 |
| 主要结果 | 主结果图 / 结果表 |
| 真实世界实验 | 机器人实拍图 / 项目页实验图 |
| 速度与工程价值 | 延迟图 / 效率图 / 消融图 |

## arXiv 图的选择建议

默认不需要把每一张图都塞进文章。推荐只保留 3 到 5 张：
- Figure 1：通常是总览图，优先级最高
- 一张架构图：解释方法设计
- 一张主结果图：承接实验表现
- 一张真实世界图：增强直观感
- 一张效率或消融图：只有正文讨论到再插

## 图注写法

不要只把论文原 caption 原封不动粘进去。

推荐写法：
1. 先说明图里展示了什么
2. 再说明这张图为什么值得看
3. 最后点出它对应正文哪一段观点

示例：

```markdown
![Figure 2：Fast-WAM 模型架构](https://arxiv.org/html/2603.16666v1/x2.png)

*图 2：Fast-WAM 在训练时联合建模视频与动作，但推理时只保留动作生成链路。这张图正好对应正文里“为什么它能兼顾世界建模收益与低时延”的那一段。*
```

## 什么时候用浏览器截图

仅在以下情况使用：
- 项目页只有整页视觉叙事，没有可直接引用的独立图片
- 用户明确要“头图风格”的页面截图
- 需要从视频播放器、canvas 或交互页面中截取内容

如果没有浏览器环境，不要因此阻塞文章交付。先用 arXiv 图完成主文稿。

## Fast-WAM 案例沉淀

Fast-WAM 这次验证下来的经验：
- 真正可复现的正文配图，不需要 Playwright 才能完成
- 直接引用 arXiv HTML 的官方图片 URL，就能得到稳定可预览的图文结果
- 更值得固化的是“选哪几张图、放在哪一段、图注怎么写”，而不是“必须截图”本身
