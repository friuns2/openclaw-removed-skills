# 第19章 配置与变更管理

## 背诵重点

1. 配置管理目标：①所有配置项能够被识别和记录②维护配置项记录的完整性③为其他管理过程提供有关配置项的准确信息④核实有关信息系统的配置记录的正确性并纠正发现的错误⑤配置项当前和历史状态得到汇报⑥确保信息系统的配置项的有效控制和管理
2. 配置项定义：是信息系统组件或与其有关的项目，包括软件、硬件、各种文档、变更请求、服务、服务器、环境、设备、网络设施、台式机、移动设备、应用系统、协议、电信服务等
3. 配置项状态：草稿【0.YZ】、正式【X.Y】、修改【X.YZ】；配置项刚建立时为草稿，通过评审后变为正式，若更改配置项则变为修改，修改完毕并重新通过评审时又变为正式
4. 配置项版本号：草稿状态格式为0.YZ（YZ取值01~99），正式状态格式为X.Y（X取值1~9，Y取值0~9，第一次成为正式文件时版本号为1.0），修改状态格式为X.YZ（修改时只增大Z值，修改完毕状态成为正式时将Z值设置为0增加X.Y值）
5. 配置基线：由一组配置项组成，这些配置项构成一个相对稳定的逻辑实体；基线中的配置项被"冻结"不能再被任何人随意修改，对基线的变更必须遵循正式的变更控制程序；基线通常对应于开发过程中的里程碑
6. 配置库三种类型：开发库（也称动态库、程序员库或工作库，可以任意的修改）、受控库（也称主库，包含当前的基线加上对基线的变更，可以修改但需要走变更流程）、产品库（也称静态库、发行库、软件仓库，包含已发布使用的各种基线的存档，一般不再修改，真要修改的话需要走变更流程）
7. 配置管理相关角色：变更控制委员会CCB（决策层，负责审批重大变更请求）、配置管理负责人（管理层，负责管理和决策整个项目生命周期中的配置活动）、配置管理员（执行层，负责在整个项目生命周期中进行配置管理的主要实施活动）、配置项负责人（专员层，确保所负责的配置项的准确和真实）
8. 配置管理日常管理活动：制订配置管理计划、配置项识别、配置项控制、配置状态报告、配置审计、配置管理回顾与改进等；口诀：计时制，状态审计会改
9. 配置审计：包括功能配置审计（审计配置项的一致性，验证配置项的实际功效是否与其需求一致）和物理配置审计（审计配置项的完整性，验证配置项的物理存在是否与预期一致）
10. 变更管理流程：①变更申请→②对变更的初审→③变更方案论证→④变更审查→⑤发出通知并实施→⑥实施监控→⑦效果评估→⑧变更收尾；口诀：提出、初审、论证、审查、通知、实施、监控、效果评估、收尾

## 补充说明

**配置项类型：**
- 基线配置项：所有的设计文档和源程序，向开发人员开放读取的权限
- 非基线配置项：项目的各类计划和报告，向PM、CCB及相关人员开放
- 所有配置项的操作权限应由CMO（配置管理员）严格管理

**配置管理关键成功因素：**
1. 所有配置项应该记录
2. 配置项应该分类
3. 所有配置项要进行编号
4. 应该定期对配置库或配置管理数据库中的配置项信息进行审计
5. 每个配置项在建立后，应有配置项负责人负责
6. 要关注配置项的变化情况
7. 应该定期对配置管理进行回顾
8. 能够与项目的其他管理活动进行关联
口诀：记分联号，变顾审责

**配置管理计划主要内容：**
1. 配置管理的目标和范围
2. 配置管理活动（配置项标识、配置项控制、配置状态报告、配置审计、发布管理与交付等）
3. 配置管理角色和责任安排
4. 实施这些活动的规范和流程
5. 负责实施这些活动的人员或团队
6. 配置管理信息系统的规划
7. 配置管理的日常事务
8. 计划的配置基准线、重大发布、里程碑

**配置项识别活动：**
1. 确定配置项范围
2. 确认和记录配置项属性
3. 为配置项定义标识符
4. 确定配置基准线
5. 确定配置结构
6. 确定配置项命名规则

**配置项控制流程（变更控制流程）：**
1. 变更申请：相关人员填写变更申请表，提交给CCB
2. 变更评估：CCB负责组织对变更申请进行评估并确定变更对项目的影响、变更的内容是否必要、变更的范围是否考虑周全、变更的实施方案是否可行、变更工作量估计是否合理，CCB决定是否接受变更
3. 通告评估结果：CCB把关于每个变更申请的批准、否决或推迟的决定通知受此处置意见影响的每个干系人
4. 变更实施：项目经理组织修改相关的配置项，并在相应的文档、程序代码或配置管理数据中记录变更信息
5. 变更验证与确认：项目经理指定人员对变更后的配置项进行测试或验证，项目经理应将变更与验证的结果提交给CCB，由其确认变更是否已经按要求完成
6. 变更的发布：配置管理员将变更后的配置项纳入基线，将变更内容和结果通知相关人员，并做好记录

**基于配置库的变更控制：**
1. 将待升级的基线从产品库中取出，放入受控库
2. 程序员将欲修改的代码段从受控库中检出，放入自己的开发库中进行修改，代码被Checkout后即被"锁定"
3. 程序员将开发库中修改好的代码段检入受控库，Checkin后，代码的"锁定"被解除
4. 软件产品的升级修改工作全部完成后，将受控库中的新基线存入产品库中

**配置状态报告内容：**
1. 每个受控配置项的标识和状态
2. 每个变更申请的状态和已批准的修改的实施状态
3. 每个基线的当前和过去版本的状态以及各版本的比较
4. 其他配置管理过程活动的记录

**配置审计作用：**
1. 防止向用户提交不适合的产品
2. 发现不完善的实现
3. 找出各配置项间不匹配或不相容的现象
4. 确认配置项已在所要求的质量控制审核之后纳入基线并入库保存
5. 确认记录和文档保持着可追溯性

**配置审计场景：**
1. 实施新的配置库或配置管理数据库之后
2. 对信息系统实施重大变更前后
3. 在一项软件发布和安装被导入实际运作环境之前
4. 灾难恢复之后或事件恢复正常之后
5. 发现未经授权的配置项后
6. 任何其他必要的时候

**变更管理原则：**
1. 基准管理：基准是变更的依据
2. 变更控制流程化：所有变更都必须遵循这个控制流程
3. 明确组织分工：至少应明确变更相关工作的评估、评审、执行的职能
4. 评估变更的可能影响
5. 妥善保存变更产生的相关文档

**变更分类：**
1. 根据变更性质：重大变更、重要变更、一般变更（通过不同审批权限控制）
2. 根据变更的迫切性：紧急变更、非紧急变更（通过不同变更处理流程）

**变更管理角色：**
1. 项目经理：响应变更提出者的需求，评估变更对项目的影响及应对方案，将需求由技术要求转化为资源需求，并据评审结果实施
2. 变更管理负责人（变更经理）：负责整个变更过程方案的结果，负责变更管理过程的监控，协调相关的资源，制定变更计划和日程安排
3. 变更请求者：负责记录与提交变更请求单，提交初步的变更方案和计划，初步评价变更的风险和影响
4. 变更实施者：按照实施计划实施具体的变更任务
5. 变更顾问委员会：负责对重大变更行使审批，提供专业意见和辅助审批，紧急变更时行使审批权限

**变更流程各环节：**
1. 变更申请：变更提出应当及时以正式方式进行，并留下书面记录
2. 对变更的初审：确认变更的必要性，格式校验完整性校验，在干系间就提出供评估的变更信息达成共识
3. 变更方案论证：对变更请求是否可实现进行论证，将变更请求由技术要求转化为资源需求
4. 变更审查：包括客户、相关领域的专业人士等，通常采用文档会签形式
5. 发出通知并实施：变更评审通过后意味着基准的调整，同时确保变更方案中的资源需求及时到位
6. 实施监控：变更实施的过程监控，通常由项目经理负责基准的监控
7. 效果评估：评估依据是项目的基准，结合变更的目标评估变更所要达到的目的是否已达成
8. 变更收尾：判断发生变更后的项目是否已纳入正常轨道

**版本发布前准备工作：**
1. 进行相关的回退分析
2. 备份版本发布所涉及的存储过程、函数等其他数据的存储及回退管理
3. 备份配置数据
4. 备份在线生产平台接口、应用、工作流等版本
5. 启动回退机制的触发条件
6. 对变更回退的机制职责的说明

**回退步骤：**
1. 通知相关用户系统开始回退
2. 通知各关联系统进行版本回退
3. 回退存储过程等数据对象
4. 配置数据回退
5. 应用程序、接口程序、工作流等版本回退
6. 回退完成通知各周边关联系统
7. 回退后进行相关测试
8. 通知用户回退完成

**软件文档三类：**
1. 开发文档：描述开发过程本身（可行性研究报告和项目任务书、需求规格说明、功能规格说明、设计规格说明、开发计划、软件集成和测试计划、质量保证计划、安全和测试信息）
2. 产品文档：描述开发过程的产物（培训手册、参考手册和用户指南、软件支持手册、产品手册和信息广告）
3. 管理文档：记录项目管理的信息（开发过程的每个阶段的进度和进度变更的记录、软件变更情况的记录、开发团队的职责定义、项目计划、项目阶段报告、配置管理计划）

**文档质量四级：**
1. 最低限度文档（1级）：适合开发工作量低于一个人月的开发者自用程序
2. 内部文档（2级）：可用于没有与其他用户共享资源的专用程序
3. 工作文档（3级）：适合于由同一单位内若干人联合开发的程序
4. 正式文档（4级）：适合那些要正式发行供普遍使用的软件产品

**记忆口诀：**
- 配置管理关键成功因素：记分联号，变顾审责
- 配置管理日常活动：计时制，状态审计会改
- 变更管理流程：提出、初审、论证、审查、通知、实施、监控、效果评估、收尾
