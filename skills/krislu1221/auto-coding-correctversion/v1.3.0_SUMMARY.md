---

## 🎯 问题起源

用户质疑：
> "你一个 Hello world 需要 98 秒？"

**根本原因**:
- ❌ 所有任务走相同流程（分析→实现→测试→反思→修复→交付）
- ❌ 简单任务也进行深度反思和多轮迭代
- ❌ 没有前置的复杂度评估机制

---

## ✅ 解决方案

### 架构创新

引入**前置需求拆解器**，智能判断任务复杂度：

```
用户输入 → 需求拆解器 → 复杂度评估 → 执行计划 → Worker
           (独立模块)     (1-10 分)     (步骤配置)
```

### 核心改进

| 改进点 | 优化前 | 优化后 |
|--------|--------|--------|
| 复杂度判断 | 无 | ✅ 前置拆解器 (1-10 分) |
| 执行流程 | 固定 7 步 | ✅ 动态调整 |
| Hello World | 98 秒 | **15 秒** ⚡ 6.5x |
| 复杂任务 | 98 秒 (不够) | **210 秒** 🔍 更充分 |

## 📦 需求拆解器设计

### 复杂度分级

| 等级 | 分数 | 执行模式 | 步骤 | 跳过 | 预计耗时 |
|------|------|----------|------|------|----------|
| **simple** | 1-3 | quick | 分析→实现→基础检查→交付 | 测试、反思、修复 | 15 秒 |
| **medium** | 4-6 | standard | 分析→实现→测试→交付 | 深度反思 | 75 秒 |
| **complex** | 7-10 | thorough | 完整流程 | 无 | 210 秒 |

### 判断逻辑

```python
# 关键词匹配
SIMPLE_KEYWORDS = ["hello", "简单", "测试", "示例", "demo", "打印"]
COMPLEX_KEYWORDS = ["系统", "完整", "生产", "优化", "重构", "架构"]

# 判断规则
if 复杂关键词 >= 2:
    return "complex"
elif 简单关键词匹配:
    return "simple"
elif 复杂关键词 >= 1:
    return "complex"
else:
    return "medium"

# 复杂度评分 (1-10 分)
score = 子任务数量 (0-3) + 子任务平均复杂度 (0-3) + 
        任务描述长度 (0-2) + 依赖关系 (0-2)
```

### 核心代码

#### 拆解器初始化

```python
# worker.py
def __init__(self, use_decomposer: bool = True):
    # 初始化需求拆解器
    if use_decomposer and DECOMPOSER_AVAILABLE:
        self.decomposer = RequirementDecomposer(llm_client=self.llm)
        print(f"✅ 需求拆解器已初始化")
```

#### 执行流程调整

```python
async def run(self, task: str):
    # 0. 需求拆解 (新增)
    if self.decomposer:
        result = self.decomposer.decompose(task)
        self.decomposer.print_plan(result)
        self._apply_decomposition_result(result)  # 动态调整
    
    # 1-7. 原有流程 (根据拆解结果调整)
    ...

def _apply_decomposition_result(self, result):
    plan = result.execution_plan
    
    # 调整工作模式
    if plan.mode == "quick":
        self.mode = WorkMode.QUICK
    
    # 调整最大迭代次数
    self.max_iterations = plan.max_iterations
    
    # 调整反思深度
    if plan.reflection_level == "surface":
        self.reflect_depth = "basic"
```
