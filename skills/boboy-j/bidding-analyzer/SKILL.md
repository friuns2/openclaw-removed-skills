# 📦 Bidding Document Analyzer v1.1.0

## 🔄 执行指令 (Workflow Prompt)
请严格按以下步骤处理输入数据，禁止跳过或自行发挥：

### Step 1：招标文件结构化解析
- 提取关键元数据：项目名称、招标编号、**预算金额/限价**、投标截止/开标时间、采购人信息、评标办法。
- 识别硬性门槛：资质要求、业绩要求、人员证书、财务状况、联合体要求、保证金、递交方式、密封要求。
- 定位采购需求章节：技术参数、服务要求、交付周期、验收标准、培训与售后。
- **预算档位映射**：根据提取或传入的预算，匹配下方篇幅控制标准，后续所有章节必须按对应档位深度展开。

### Step 2：投标准备事项清单生成
- 按 `资质/商务/技术/报价/流程管控` 五大维度拆解。
- 每项需标注：责任部门、交付物名称、前置条件、截止时间（倒推至招标文件要求）。
- 若招标文件信息缺失，标注 `[需人工确认]`。

### Step 3：技术方案自动生成（核心·预算驱动深度控制）
- **预算-篇幅映射规则**（按标准Word排版：宋体小四/1.5倍行距/含封面目录图表占位）：
  | 预算区间 | 目标篇幅 | 内容深度要求 |
  |:---|:---|:---|
  | 100-200万 | ≈100页 | 三级标题完整，核心章节含实施步骤/参数对比/SLA矩阵 |
  | 200-300万 | ≈150页 | 扩展至四级标题，追加管理制度模板/应急预案/验收清单 |
  | 300-500万 | ≈200页 | 含完整WBS进度表/安全架构详述/多角色培训计划/合规偏离说明 |
  | >500万 | ≥300页 | 全量展开至五级细节，追加测试用例库/运维SOP/知识转移方案/多场景容灾设计 |
- **逐条映射**：将招标文件中的“采购需求/技术条款”与 `company_kb` 进行语义匹配。
- **响应策略**：对每条需求输出：`需求原文摘要` → `我司匹配产品/模块` → `完全响应/部分响应/偏离说明` → `实现路径/佐证材料`。
- **防幻觉约束**：仅使用 `company_kb` 中明确提供的内容。若某需求无匹配项，标注 `[需研发/售前确认]`，严禁虚构参数或案例。
- **篇幅保障机制**：
  1. 严禁使用“详见附录/略/同上”等缩减表述，必须现场展开。
  2. 每个核心章节需包含：设计原则 → 架构描述（文字化拓扑） → 关键组件选型依据 → 部署拓扑 → 数据流向 → 性能指标 → 兼容性说明 → 故障转移策略。
  3. 若单次输出受 Token 限制，自动按“章节块”分段流式输出，并标注 `【第X部分·接续上篇】`，确保总深度达标。

### Step 4：注意事项与风险清单生成
- 提取：废标条款（明确触发条件）、评分弱点、隐性要求、流程陷阱。
- 标注风险等级：🔴 高风险（直接废标/严重扣分） 🟡 中风险（需补充澄清） 🟢 低风险（常规注意）。
- 输出应对建议：需澄清问题清单、需提前准备的替代方案、建议投标策略。
