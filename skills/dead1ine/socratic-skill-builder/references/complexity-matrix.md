# 复杂度评估矩阵

## 评估量表（每项 0~2 分）

| 维度 | 0 分 | 1 分 | 2 分 |
|------|------|------|------|
| **步骤数量** | 1 个步骤 | 2~3 个步骤 | 4 个以上步骤 |
| **外部依赖** | 无依赖 | 1 个 API/工具 | 2 个以上 API/工具 |
| **分支判断** | 无条件分支 | 1~2 个 if/else | 复杂判断、多路径分支 |
| **容错需求** | 不需要容错 | 简单 try/catch | 复杂异常处理、重试、断点续传 |
| **状态管理** | 无状态 | 简单上下文（记住1~2个变量） | 多轮对话、跨会话状态 |
| **输入复杂度** | 纯文本 | 结构化数据（JSON/列表） | 非结构化或混合输入 |
| **输出复杂度** | 纯文本 | 文件或结构化数据 | 多文件/多格式/直接执行动作 |

**总分：0+0+0+0+0+0+0 = 0（最低）~ 14（最高）**

---

## 复杂度等级与对话轮数对照表

| 总分 | 等级 | 对话轮数 | 典型特征 | 技能示例 |
|------|------|---------|---------|---------|
| 0~3 | 🌱 简单 | 2~3 轮 | 单步操作、无分支、文本输入输出 | 文本格式化、计算器 |
| 4~6 | 🌿 中等 | 3~5 轮 | 2~3步骤、1个API、有简单分支 | 天气查询、邮件格式化 |
| 7~9 | 🌳 复杂 | 5~7 轮 | 多步骤、多API、复杂分支、需要容错 | 爬虫+摘要、数据清洗 |
| 10~12 | 🌲 超复杂 | 7~9 轮 | 长链路、多系统集成、状态机、需架构设计 | 全栈自动化、推荐系统 |
| 13~14 | 🔮 极限 | 9+ 轮 | 未解决的技术问题、高度不确定性 | 全新领域、实验性项目 |

## 复杂度快速判断法

### 30秒快速判断问题（闭眼直觉）

1. 「这个技能，用户说一句话，**能直接得到答案**吗？」
   - ✅ 能 → 简单
   - ❌ 不能 → 继续问

2. 「这个技能，用户说一句话，需要**超过3个步骤**才能完成吗？」
   - ❌ 不需要 → 中等
   - ✅ 需要 → 继续问

3. 「这个技能涉及**两个以上外部系统**或**需要持久化状态**吗？」
   - ❌ 否 → 复杂
   - ✅ 是 → 超复杂/极限

## 复杂度对话示例

### 🌱 简单技能（2轮对话）

**需求**：「帮我做个技能，把英文日期格式转成中文」
```
问：「只需要转日期格式，不需要处理时区或日历计算对吧？」
答：「对，固定输入格式，只转文字」
→ SKILL.md 直接生成，2轮结束
```

### 🌳 复杂技能（5~6轮对话）

**需求**：「帮我做个技能，爬取竞争对手网站每周价格变化，发邮件给我」
```
问（第一轮）：「竞争对手网站结构稳定吗？登录保护？」
答：「结构不固定，有反爬」
问（第二轮）：「邮件是发送到固定邮箱还是需要每次指定？」
答：「固定自己的邮箱」
问（第三轮）：「'价格变化'是只报告涨价/降价，还是包含原价、新价、变化幅度？」
答：「最好包含三个」
问（第四轮）：「如果某周网站改版爬不到，邮件要报告'数据缺失'对吧？」
答：「对，宁可报错也不要错误数据」
→ 5轮后生成完整 SKILL.md
```

## 动态复杂度调整

有些需求在对话过程中**复杂度会改变**：

| 情况 | 调整方向 | 应对策略 |
|------|---------|---------|
| 用户说「随便，不重要」| 降级 | 跳过细节追问，快速生成 |
| 用户说「这个必须支持」| 升级 | 增加对应维度的深度追问 |
| 发现未预判的技术障碍 | 显著升级 | 明确告知，延长对话轮数 |
| 用户需求明显超出能力边界 | 降至最低 | 明确边界，重新定义问题 |

> **调整时告知用户**：「根据你的回答，这个需求的复杂度从**中等**升级到**复杂**了，我需要多问几个问题确保做对。」
