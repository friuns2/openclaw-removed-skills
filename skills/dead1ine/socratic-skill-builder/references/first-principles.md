# 第一性原理分析框架

## 什么是第一性原理拆解？

「从最基本的不可再分的真理出发，一层层往上构建。」

应用于技能设计：把一个技能**拆解到无法再拆解的最小单元**，然后重新审视：「这些单元都是必要的吗？」

---

## 技能的第一性原理：三层拆解模型

### 第1层：目的（Why）— 这个技能为什么存在

**核心问题链：**
```
用户现在的**痛苦**是什么？
  ↓
这个痛苦**频率**有多高？（每天？每周？偶尔？）
  ↓
如果没有这个技能，**成本**是多少？（时间/金钱/错误率）
  ↓
这个技能**必然存在**的理由是什么？
```

**检验问题：**
> 「如果这个技能只能做到**最核心的一件事**，那件事是什么？」
> 「这件事是否**离了技能就无法完成**？」

---

### 第2层：机制（What/How）— 技能实际做什么

**核心问题链：**
```
输入的**最小信息单元**是什么？（一个关键词？一篇文章？一个文件？）
  ↓
处理的**最小操作单元**是什么？（搜索/读取/转换/判断/发送）
  ↓
输出的**最小结果单元**是什么？（一条文本？一个文件？一个动作？）
  ↓
这些单元之间的**因果链**是什么？
```

**检验问题：**
> 「如果把某个步骤**去掉**，技能还能正常运行吗？」
> 「有没有哪个步骤是**假装的**——看起来在工作但实际不影响结果？」

---

### 第3层：边界（Limits）— 技能不做什么

**核心问题链：**
```
技能的**最简单成功情况**是什么？（Happy Path）
  ↓
技能的**最简单失败情况**是什么？（Minimum Failure）
  ↓
技能**绝对不应该处理的输入**是什么？
  ↓
技能的**生命周期**是什么？（临时解决方案还是长期工具？）
```

---

## 第一性原理拆解练习模板

**问题A：「帮我做一个新闻摘要技能」**

```
拆解：
❌ 不需要：新闻来源（用户应该自己找）
❌ 不需要：多语言翻译（除非用户要求）
❌ 不需要：情感分析（不是核心）
✅ 只需要：输入URL → 提取正文 → 生成3句话摘要

结论：本质是「URL → 摘要文本」，而不是「新闻工作流」
```

**问题B：「帮我做个小红书文案生成器」**

```
拆解：
❌ 不需要：生成图片（那是另一个技能）
❌ 不需要：SEO关键词优化（不是核心用户痛点）
❌ 不需要：多平台发布（那是后续扩展）
✅ 核心：产品特点描述 → 情绪化文案 + Hashtag

结论：本质是「卖点 → 情绪化文字」，核心在「翻译」而非「生成」
```

## 降本增效检验表

完成拆解后，用这张表检验：

| 检验项 | 问题 | 如果「否」，简化方案？ |
|--------|------|----------------------|
| 必要性 | 这个步骤用户**真的需要**还是**觉得需要**？ | 移除或标记为可选 |
| 不可分 | 这个步骤还能再拆吗？ | 拆解为独立子步骤 |
| 可测试 | 每一步完成后，能**明确判断**对错吗？ | 增加验证逻辑 |
| 可逆性 | 如果这一步错了，能**回退**吗？ | 增加回退机制 |
| 价值比 | 这一步占用了多少**开发成本**，带来了多少**用户价值**？ | 成本>价值则移除 |

## 常见第一性原则错误

1. **功能蔓延**：在核心周围堆砌「也许有用」的功能
2. **技术牵引**：先想「用什么技术实现」而不是「用户真正要什么」
3. **完美主义**：在用户根本不会用的细节上过度设计
4. **假设需求**：假设用户需要某功能而未经验证
5. **重复造轮子**：做一个和现有工具高度重叠的技能

> **修正问题**：> 「有没有一个**现有工具**已经做到 80% 了？」
