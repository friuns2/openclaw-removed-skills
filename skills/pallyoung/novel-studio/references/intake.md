# Intake

## 1. Goal

Capture the user’s current preference, constraints, and creative intent after the initial discovery discussion begins.

This stage should narrow and clarify what the user actually wants, not guess it.

---

## 2. Required information

Collect as many of these as possible:
- audience type
- genre or topic preference
- tone or style
- target platform if relevant
- target length or scale
- whether adaptation potential matters
- hard constraints or taboo elements
- user priorities

---

## 3. Optional information

If available, also collect:
- reference works
- comparable titles
- opening image or key emotional scene
- pacing preference
- romance structure preference
- whether the project should optimize for serialization, retention, or literary feel

---

## 4. Clarification rules

Ask follow-up questions when:
- the request is too vague to distinguish direction
- the user names conflicting goals
- the target audience is unclear
- the tone is unstable
- the commercial goal affects structural decisions

Do not pretend unclear direction is already usable.

---

## 5. Required output

This stage must produce a preference-capture artifact written into project files.

Recommended artifact:
- `00B_用户偏好.md`

The artifact should record:
- user preference summary
- audience preference
- tone / style preference
- platform tendency if any
- constraints / taboo elements
- user priority ranking
- unresolved questions if any

---

## 6. Completion standard

This stage is complete only if:
- user preference summary exists
- major constraints are recorded
- audience or platform orientation is usable enough to influence topic choice
- the output is written to a file, not only kept in chat

---

## 7. Do not advance if

Do not advance to formal market recommendation if:
- user preferences are still too ambiguous to influence recommendation
- key constraints are missing
- preference capture exists only in chat and not in project files

---

## 8. Rollback condition

Return to intake if:
- user changes preference significantly during discovery
- recommendation work reveals unresolved conflicts in user goals

---

## 9. Quality bar

A valid intake result is:
- usable
- preference-specific
- constraint-aware
- written down

Invalid output includes:
- vague preference guesses
- no written artifact
- no record of actual user priorities
