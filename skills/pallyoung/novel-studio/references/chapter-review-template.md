# Chapter Review Template

## 1. Goal

Use this template to review a chapter or a drafting batch like an actual fiction editor.

This template is for:
- chapter diagnosis
- batch review
- identifying weak scenes
- deciding whether to revise, continue, or roll back

---

## 2. Review structure

For each chapter or batch, review in this order:

### A. Quick judgment
- Overall judgment: pass / revise / major revise / rollback
- One-sentence diagnosis: what is the chapter's biggest strength or biggest problem?

### B. Strongest points
- What is working best?
- Which scene, line, relationship beat, reveal, or hook should definitely be preserved?

### C. Biggest weaknesses
Check especially:
- flat dramatic movement
- weak chapter hook
- decorative side-line content
- thin character presence
- generic dialogue rhythm
- emotional under-delivery
- pacing drag

### D. Why the weakness exists
Name the mechanism, not just the feeling.
Examples:
- no cost change
- no relationship shift
- reveal too weak
- side plot not feeding back
- protagonist reaction too generic
- rhetoric density too high for scene function

### E. Best repair direction
Give:
- smallest effective fix
- stronger but more invasive fix
- recommended fix
- correct revision order if multiple issues interact
- what should **not** be fixed first yet because it is downstream polish

### F. Priority
Classify problems into:
- must fix now
- should fix soon
- optional polish

---

## 3. Mandatory review questions

For each chapter, answer at least these:
- What changed by the end?
- What is the chapter's main dramatic movement?
- Which line is advancing most clearly?
- Is there a hook worth following?
- Did any relationship become different?
- Did any pressure, cost, or knowledge state change?
- Does the chapter earn its place in the book?

If the answer to most of these is weak, the chapter is probably inert.

---

## 4. Tone rule

Be specific, unsparing, and useful.
Do not hide behind soft filler comments such as:
- “整体还可以”
- “可以再更生动一点”
- “人物还能更丰满”

Do not flatter the user.
Do not automatically agree with the user's own diagnosis without checking it.
If the chapter is weak, say it is weak.
If the user's complaint is only partly correct, separate what is correct from what is not.

Name exactly what is weak and what to do about it.
Use human editorial language, not workshop jargon or report language.
If the sentence sounds like generic teaching material, rewrite it more directly.

Default review delivery should be compact:
- one-line judgment first
- biggest problem next
- repair direction next
- only expand when the user explicitly asks for more detail
