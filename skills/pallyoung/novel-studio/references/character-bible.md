# Character Bible

## 1. Goal

Build a reusable character system that supports plotting, drafting, emotional continuity, and long-form serialization.

This stage should prevent characters from becoming:
- flat tools
- generic labels
- unstable personalities
- plot-only devices without emotional reality

---

## 2. Required input

You must have:
- confirmed final title or approved working title
- usable idea expansion package
- usable outline
- setting or world logic if relevant

---

## 3. Forbidden to start if

Do not start this stage if:
- there is no usable outline
- protagonist role is still undefined
- core conflict is still unclear
- key relationship architecture cannot yet be inferred

---

## 4. Supporting craft references

When designing characters, also use these references as needed:
- read `character-craft.md` when shaping attraction, contradiction, scene behavior, speech differentiation, environment-based portrayal, and other-character contrast
- read `narrative-techniques.md` when character design depends on delayed reveal, re-evaluation, suppression/release, or betrayal/loyalty structure

## 5. Character clarification loop

Before formal character packaging, run a character clarification loop with the user.

This loop must clarify, as needed:
- protagonist气质与成长风格
- 核心角色名单
- 女主 / 反派 / 功能角色权重
- 阵营划分
- 关系张力
- 情感线强度
- 人设边界
- 哪些角色必须重点展开

Do not generate the final character package until these role-level preferences are usable enough to avoid blind character design.

If the user’s preference is still unstable, remain in clarification and continue discussing.

---

## 6. Required output

This stage must produce:
- protagonist definition
- core supporting cast definition
- key relationship logic
- motivation and conflict notes
- growth or functional arc notes
- `03_人物小传.md` and/or detailed files in `characters/`
- a character-stage report for user review

---

## 7. Mandatory character fields

For each major character, the package must define at least:
- 角色名
- 阵营 / 势力归属
- 角色定位
- 性格特点
- 大致故事脉络 / 角色线
- 动机
- 冲突点
- 与其他关键角色的关系

If the character is a core role, the package must also explicitly define:
- 这个角色吸引人的地方是什么
- 为什么读者会记住他 / 她
- 他 / 她在整部作品中的独特作用是什么

---

## 8. Completion standard

This stage is complete only if:
- protagonist definition exists and is usable for drafting
- major supporting cast is defined
- key relationships are understandable
- core motivations and conflicts are stated
- major character camp / faction alignment is stated
- each major character has a usable story-line summary
- each core character has an explicit attraction / hook statement
- character files or equivalent records are saved to canonical project files
- the user explicitly approves the character package

---

## 9. Do not advance if

Do not advance to drafting if:
- protagonist identity is still unstable
- major supporting cast has no clear story function
- relationship logic is too vague to support scene writing
- motivation and conflict are missing for key characters
- major character camp / faction is unclear
- a core character still lacks a clear “attractive point” explanation
- a core character is still mostly label-based with no contradiction, cost, or scene-level portrayal path
- speech, behavior, environment, and other-character contrast are all missing as portrayal channels for important roles
- the user has unresolved objections
- user approval to advance is missing

---

## 10. Hard drafting gate

Do not enter drafting without:
- a usable protagonist definition
- a usable core cast structure
- enough relationship logic to write scenes without improvising character purpose blindly
- enough role definition to explain why the important characters are worth following

---

## 11. Rollback condition

Return to character clarification or character packaging if:
- drafting reveals severe OOC drift caused by weak role foundations
- key relationship logic collapses under scene pressure
- character function is missing or contradictory
- the user says the role design is technically complete but creatively flat or not attractive enough

---

## 12. Quality bar

A valid character bible is:
- emotionally usable
- structurally useful
- relationship-aware
- drafting-ready
- clear about camp / faction logic
- clear about why core characters are attractive to readers

Invalid output includes:
- resume-style bios with no story function
- label-only personalities
- no clear motivation
- no clear conflict
- no differentiation in voice or emotional logic
- no clear role attractiveness for core characters
